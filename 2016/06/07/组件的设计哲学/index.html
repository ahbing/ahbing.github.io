组件的设计哲学

发表于 2016-06-07 | 分类于 react |

组件化开发有一个重要的目的就是实现组件的复用，但组件是不完全独立的，比如组件的数据， UI 样式的表现，在不同的场景下，可能是不同的。所以如何保持组件在复杂场景下的可复用性，是需要经过一番设计的。

在 react 官方文档中也明确提到，组件应尽可能的保持无状态化，组件所需要用到的数据从父组件通过 props 传进来，父组件传什么数据，子组件就渲染什么数据。这就在一定程度上保证了组件的可复用性。

组件的 UI 相关的 state 的管理， react 官方也给出了一个方案，首先列出组件所有的用到的管理 UI 的 state，然后将 所有的 state 的交给共同的祖先组件管理，如果没有共同的祖先组件，创建一个组件来包含所有的子组件，这种组件类似一个容器，姑且把它叫做容器组件吧。

在简单的应用里，按照上面的思路，可以合理的编写可复用的组件，这里说的简单是指组件嵌套不是很深的情况，比如官方提供的 todo ，组件嵌套两三层，只需一个容器组件。

拆分容器组件

上文提到， 之所以创建容器组件的原因是，我们需要一个组件，来专门管理其下所有子组件的状态。本来创建的 容器组件是为了统一 state 入口，方便 state 的管理。 但是，在一个庞大的应用中，组件的嵌套很深，容器组件管理的 state 很多很多。已经无法满足当初设立的目的了。

怎么办？

一级容器组件 (维护一级组件的 state)
    某个视图组件（渲染一级组件的 state）

        二级容器组件 (维护二级组件的 state)
            某个视图组件 （渲染二级组件的 state）

                三级容器组件 （维护三级组件的 state）
                    某个视图组件 （渲染三级组件 state）

                三级容器组件
                    某个视图组件

                三级容器组件
                    某个视图组件    
                    某个视图组件

        二级容器组件
            ...

如上， 当细分容器之后，每个容器组件只负责维护一部分的 state。

组件职责

Redux 作者 Dan Abramov 写过一篇文章 Presentational and Container Components 把这两种组件的职责总结的很完善。

以下基于自己的理解和思考总结的。

视图组件

  • 关心组件样式。
  • 视图组件可以包含容器组件和视图组件，但是有自己的 DOM 结构和 样式。
  • 通常通过 this.props.children 来嵌套容器组件。
  • 不依赖应用的部分，比如 Flux 中的 actions 和 stores
  • 不去请求数据， 也不去修改 state
  • 通过 props 来接收数据和函数
  • 尽可能自身不维护 state （如果一定要有，也只能是 UI state，决不能是数据）
  • 如果组件不需要 state ，生命周期的钩子的化，应尽量写成 函数式组件

容器组件

  • 关心组件需要哪些内容，如何工作。
  • 可以包含容器组件和视图，但除了一些包装的 divs 不应该任何的 DOM 标签，和任何样式
  • 给子组件提供数据和函数，比如 从服务器获取的数据， 或者 Flux的 actions 的回调。
  • 通常用高阶函数生成，比如 React Redux 的 connect()。

小结

  • 把组件分成容器组件和视图组件，组件的职责更加清晰，应用的结构也更加明朗。
  • 解耦数据和结构，使得视图组件的复用性更好。
